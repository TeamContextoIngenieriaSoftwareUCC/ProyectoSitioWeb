Crisis

La Crisis del Software empieza a mediados de los años 60,. Esta crisis  expresaba las dificultades del desarrollo de software y el deficiente rendimiento de este frente a la gran demanda que había de software por parte de las compañías. En 1968 se realizó una primera conferencia por la OTAN donde por primera vez se habló a profundidad de la "crisis del software" y para tratar de dar una solución o avance a esta problema se presento una nueva rama de ingeniera, siendo la ingeniería de software.

La potencia computacional de las máquinas fue limitada. Por ende los programas que funcionaban en ellas eran “simples”, no tenían una metodología clara. Durante este periodo usar los lenguajes de bajo nivel para el desarrollo de Software.

Durante este Peridodo nacieron diferentes proyectos para el software pero debido a su inestabilidad se presentaban muchos errores o situación complejas, las más típicas era: retrasos, excedian la estimación de costos, el producto no era el deseado, el mantenimiento era muy complicado y las modificaciones casi ni existían, en resumen se hacía un software de muy mala calidad y cada vez se estancaba más y más.

Una de las principales causas o bien podría a ser la principal era el mal proceso de planificación, que se hacía antes de empezar el proyecto, donde se dedicaba muy poco tiempo al análisis, diseño y desarrollo y el resto del tiempo se utilizaba a la corrección de errores y mantenimiento y esto hacia que el programa surgiera con múltiples errores por la mala planificación y organización.

Aquí llegó la ingeniería de software como la disciplina que busca integrar métodos, herramientas y procesos para el desarrollo del software, aquí ya se dejaban claro procesos y pautas a seguir que minimizar, facilitar el tiempo de desarrollo, el esfuerzo y el costo, dejando así un software de calidad.

Mitos

Mito

Se tiene un libro con todos los estandares y procedimientos.

Realidad

Por contener todo ese contenido no quiere decir que contenga lo que se require saber.

Mito

Añadir programadores repondrá el tiempo perdido.

Realidad

Es programar no es un proceso mecánico. Añadir mas personas retrasara aún más el proyecto.

Mito

Cambiar es una mala idea.

Realidad

Es necesario irse adaptando a las nuevas tecnologías con el pasar del tiempo.

Mito

Una declaración de objetivos es suficiente para el desarrollo del proyecto.

Realidad

Una mala definición inicial de requirimiento es la principal causa de fallos en el software.

Mito

Los cambios en el proyecto se acomodan fácilmente debido a que el software es flexible.

Realidad

A medida que pasa el tiempo se va haciendo más y más costoso la implementacion del cambio.

Mito

Existe un lenguaje maestro donde se pueda programar de todo, y de manera totalmente eficiente.

Realidad

La diversidad en el área del desarrollo es máxima y cada cosa es mejor para su "cosa". No existe algún lenguaje maestro.

Mito

Una vez funcionando el programa el trabajo ha terminado.

Realidad

Se ha comprobado que el trabajo completado se ha realizado aun después de que se le entrega por primera vez al cliente.

Mito

Hasta que no se este ejecutando el programa no hay manera de probar su calidad.

Realidad

En la actualidad, existen tecnicas como los unit testing que sirven para probar cada parte del programa sin necesidad de estar ejecutando todos sus apartados.

Mito

La ingeniería de software obliga a realizar documentación voluminosa e innecesaria, como resultado un proceso más lento

Realidad

La ingeniería de software no es realizar documentacion, es la búsqeda de calidad, y con esta se reducen los trabajos redundantes, obteniendo un proceso más ágil.
